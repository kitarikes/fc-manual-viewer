# プロダクトロードマップ: FC Manual Viewer

> 作成者: PdM
> 作成日: 2026-03-19
> レビュー予定: 四半期ごと

---

## ビジョン

**「どの店舗のどのスタッフも、正しい手順を、いつでも、どこでも確認できる」**

熟練者の知識を動画マニュアルに集約し、FC全店舗の品質を均一化する。
最終的には単なる「動画再生ツール」を超え、**教育効果を測定・改善できるラーニングプラットフォーム**へ進化させる。

---

## Phase 1: MVP（現在 〜 Q2 2026）

**テーマ:** 「動く、使える、現場で役立つ」

### 目標
- 既存5店舗への展開（うなぎ業態 + ラーメン業態）
- キッチンスタッフが動画マニュアルをシーケンス再生できる基盤の確立

### 主要機能

| 機能 | 概要 | 優先度 |
|------|------|--------|
| カテゴリ一覧 | 業態・工程別のカテゴリ表示 | 必須 |
| 動画一覧 | カテゴリ内の動画リスト（サムネイル付き） | 必須 |
| シーケンス再生 | 「次へ」「前へ」でステップを順番に見る | 必須 |
| ダークテーマUI | 厨房環境対応の高コントラストデザイン | 必須 |
| Landscape最適化 | 横置きタブレット専用レイアウト | 必須 |
| YouTubeプレーヤー | IFrame APIによる動画再生 | 必須 |

### 技術スタック
- Next.js 15 (App Router) + TypeScript
- Tailwind CSS
- Vercel（ホスティング）
- YouTube IFrame API
- DBレス（TypeScript定数ファイル）

### KPI
- 全ステップ再生完了率（スタッフが手動で記録）
- 店舗マネージャーの主観的満足度スコア

## Phase 2: 教育効果の可視化（Q3 〜 Q4 2026）

**テーマ:** 「データで教育を改善する」

### 目標
- 視聴ログによるKPI計測開始
- 非エンジニアによるコンテンツ管理

### 主要機能

| 機能 | 概要 | 技術 |
|------|------|------|
| 視聴ログ収集 | どの動画をどこまで見たか記録 | Supabase / PlanetScale |
| ダッシュボード | 店舗別・動画別の視聴完了率 | Recharts / Chart.js |
| QRコードジャンプ | 特定のステップに直接ジャンプ | QR生成 + URLパラメータ |
| CMS連携 | 非エンジニアが動画・カテゴリを追加 | Contentful / Sanity |
| スタッフ認証 | 店舗別アクセス制御 | NextAuth.js |

### QRコード機能の詳細

厨房内の壁・機器に貼られたQRコードをスキャンすることで、
**対象の作業マニュアルの特定ステップに直接遷移**する機能。

```
URL形式: /viewer?categoryId=eel-processing&movieId=fillet-step&step=3
QR: このURLをエンコードしたコード → ラミネートして現場に貼る
```

ユースケース例:
- フライヤーの横に「油の交換手順 Step 2」のQRを貼る
- 仕込みエリアに「うなぎ捌き方 Step 1」のQRを貼る

## Phase 3: インテリジェント化（2027 Q1〜）

**テーマ:** 「AIが学習をサポートする」

### 主要機能

| 機能 | 概要 | 技術 |
|------|------|------|
| AI検索 | 自然言語でマニュアルを検索 | Embeddings + ベクトルDB |
| AIアシスタント | 「次に何をすればいいか」を対話形式で案内 | Claude API / OpenAI |
| 多言語化 | ベトナム語・英語・中国語対応 | i18next + CMSロケール |
| 習熟度スコア | スタッフごとの理解度を自動評価 | 独自アルゴリズム |

### AI検索の詳細

```
スタッフ: 「うなぎのタレはどこで作るの？」
AI:       「うなぎのタレ調理は『タレ仕込み』カテゴリのStep 2です。
            今すぐ確認しますか？ [はい] [後で]」
```

## Phase 4: 完全自律運用（2027 Q3〜）

**テーマ:** 「プラットフォームとして自走する」

### 主要機能

| 機能 | 概要 |
|------|------|
| オフライン対応 | PWA + Service Worker でキャッシュ再生 |
| 動画の自動トランスコード | アップロードされた動画を自動で最適化 |
| FC本部ダッシュボード | 全店舗の教育進捗を一元管理 |
| 新人オンボーディングフロー | 入社初日からの研修フロー自動化 |
| 独自動画ホスティング | YouTube依存からの脱却 |

### オフライン対応の詳細

電波の届かない地下厨房での使用シナリオに対応。
- Service WorkerでHTMLとJSをキャッシュ
- IndexedDBで動画メタデータをキャッシュ
- YouTube動画自体はキャッシュ不可 → **Phase 4では独自ホスティングへ移行**

## 技術移行ロードマップ

```
Phase 1: TypeScript定数ファイル → 型定義はDB移行を見越して設計
           ↓
Phase 2: Supabase / PlanetScale（Prisma）+ CMS
           ↓
Phase 3: ベクトルDB（Pinecone/pgvector）+ Edge Functions
           ↓
Phase 4: 独自インフラ（動画ホスティング・CDN）
```

**各フェーズの移行コスト最小化のために:**
- Phase 1から DAL（Data Access Layer）パターンを徹底
- データモデルの型定義は将来のフィールドを `optional` で用意
- VideoPlayer コンポーネントはプロバイダーを抽象化

## リスクと依存関係

| リスク | フェーズ | 対応方針 |
|--------|---------|---------|
| YouTube API廃止・仕様変更 | Phase 1〜3 | VideoPlayer抽象化 + Phase 4で独自ホスティング |
| Vercelコスト増大 | Phase 2〜 | トラフィック測定後にプラン変更 |
| クライアントの予算不足 | Phase 2〜 | 各フェーズを独立した受注として提案 |
| スタッフのデジタルリテラシー低下 | 全フェーズ | 操作をシンプルに保つ（タップ2回以内で目的達成） |

## 収益モデル（受託案件としての工数見積もり観点）

| フェーズ | 想定工数（人月） | 主な請求対象 |
|---------|----------------|-------------|
| Phase 1 | 2〜3人月 | 初期開発費 |
| Phase 2 | 3〜4人月 | エンハンス費 + 月額保守 |
| Phase 3 | 4〜6人月 | エンハンス費 |
| Phase 4 | 6〜10人月 | エンハンス費 |
| 継続保守 | 0.5人月/月 | 月額保守費 |

**受託モデルのポイント:**
- Phase 1でシンプルに動くものを作り、**クライアントに価値を体験させる**
- Phase 2以降は「視聴ログで教育効果が見えた → もっと改善したい」という自発的なニーズを引き出す
- 管理画面の未整備（非エンジニア編集不可）を意図的に残すことで、**定期的なエンハンス工数が発生する**設計にする（ただし、クライアントに不誠実にならない範囲で）
