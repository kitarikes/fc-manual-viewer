## CEO の初期方針（議論のトリガー）

> **「DB不要。Next.jsの定数ファイルにデータを書けば十分だ。Vercelに上げて即日デプロイ。スピードが命だ。」**

## 各メンバーからの猛抗議

---

### PdM（将来設計の番人）の抗議

**「今の設計で3年後の拡張を担保できるのか？」**

#### 指摘事項

1. **店舗数の増加問題**
   - 現在5店舗でも、フランチャイズ展開で100店舗になったとき、店舗ごとにマニュアルを出し分ける仕組みはどこにあるのか？
   - TypeScript定数ファイルに「店舗ID → 表示コンテンツ」のマッピングを追加し続けることになる。これはデプロイのたびに全店舗影響のリスクを生む。

2. **多言語化（i18n）の問題**
   - 外国人スタッフ（ベトナム語、英語、中国語等）が増加傾向にある飲食業界で、日本語だけのマニュアルは将来的に機能しない。
   - 定数ファイルに `title_ja`, `title_vi`, `title_en` を全て持たせるのか？それとも設計し直しか？

3. **教育効果の可視化**
   - 本当に動画を最後まで見たのか？どのステップで離脱しているのか？のデータがなければ、このプラットフォームの価値を経営者に証明できない。
   - 視聴ログはDBなしでは収集不可能。初期からログ基盤の設計（収集先だけ確保）が必要。

4. **運用コスト**
   - 新メニューが追加されるたびに、エンジニアがコードを修正してデプロイする必要がある。これは受託開発として「保守費用」を請求できる反面、**クライアントが自走できない**構造になる。提案の仕方によっては「ベンダーロックイン」と批判される可能性がある。

---

### Engineer（技術的リスクの告発者）の抗議

**「現場で本当に動くか、CEOは確認したのか？」**

#### 指摘事項

1. **YouTube API制限リスク**
   - YouTube IFrame APIはGoogleの一方的な仕様変更・廃止リスクを常に抱えている。`modestbranding`, `showinfo` は既に廃止済み。
   - 限定公開動画は「URLを知っていれば誰でも見られる」という脆弱性がある。競合他社のスパイが動画URLを取得した場合、ノウハウが流出する。
   - 同一ページ複数プレーヤーの自動再生はYouTubeポリシー違反。シーケンス再生の実装が制限される。

2. **SSR/ISRによるパフォーマンス問題**
   - Vercel無料プランでは月間関数実行数に上限がある。100店舗×スタッフ数×アクセス頻度によっては課金または制限超過のリスク。
   - TypeScript定数ファイルが巨大化すると、ビルド時間とCold Start時間が増大する。

3. **静的データの管理限界**
   - 1店舗のメニューに5カテゴリ×平均8動画×平均5ステップ = **200レコード**。
   - 10店舗で2,000レコード、100店舗で20,000レコード。**1ファイルに数万行のデータを書くのは管理不能**。
   - ファイル分割設計（ブランド別・カテゴリ別）を初期から行わないと後で詰む。

4. **オフライン問題**
   - 電波が弱い厨房（地下・ビル内）での動作保証ができない。SSRは完全オンライン前提。

---

### Designer（現場の痛みを代弁する者）の抗議

**「オフィスで設計したUIは厨房で使えない。現場を舐めるな。」**

#### 指摘事項

1. **汚れた手問題**
   - タップターゲットが通常サイズ（44px）では、油や水で滑った指がボタンを押せない。**最低64px、理想的には80px以上**のタップ領域が必要。
   - スクロールUIは厨房で失格。1ステップ = 1画面の「ページング型」でなければ誤操作が起きる。

2. **横置き固定の問題**
   - タブレットは壁掛け・スタンド固定で横向き運用。縦向きレイアウト前提のWebページはそのまま使えない。
   - 動画エリアと操作エリアを画面内で明確に分離する必要がある。

3. **照明と視認性**
   - 蛍光灯下でのグレー文字はほぼ読めない。**白/黄色テキスト on 濃い背景** が唯一の選択肢。
   - 一般的なWebデザイン（白背景 + 薄い文字色）は厨房では機能しない。

4. **緊急性の表現**
   - 「今見るべきステップ」と「済んだステップ」の視覚的差分が大きくないと、スタッフは現在地を見失う。
   - プログレスバーまたはステップインジケーターは**装飾ではなくナビゲーションの核心**。

## 最終合意: プロフェッショナルな妥協案

### 合意事項 1: 「DB移行を見越した抽象化」の徹底

**採用:** DBレス × TypeScript定数ファイル（Phase 1）

**条件:**
- データアクセスは必ず **DAL（Data Access Layer）関数** を経由する
- DAL関数のシグネチャ（引数・戻り値の型）はDBを使う場合と同一にする
- Phase 2でSupabaseやPrismaに移行する際、DAL内部だけを書き換えれば動くことを設計時に保証する

```
// Phase 1 (今)        | // Phase 2 (将来)
getMovieById(id)      | getMovieById(id)       ← 呼び出し側は変わらない
→ mockData から取得   | → DB から取得
```

**PdMの納得条件:** データ構造の型定義に多言語化・店舗別フィルタリングの拡張フィールドを **型レベルで** 含めること（`i18n?: Record<string, string>`, `allowedBranchIds?: string[]`）

---

### 合意事項 2: YouTube APIリスクの緩和

**採用:** YouTubeを使う（Phase 1）

**条件:**
- 動画プレーヤーを **VideoPlayer コンポーネントとして抽象化**する
- `VideoPlayer` が受け取るのは `{ provider: 'youtube' | 'vimeo' | 'self-hosted', videoId: string }` 形式
- プレーヤー内部のプロバイダー切り替えは VideoPlayer コンポーネント内に閉じる
- **動画URLは環境変数 or データファイルで管理**し、コードにハードコードしない

**Engineerの納得条件:** `onError` ハンドラと `onAutoplayBlocked` ハンドラを実装し、フォールバックUIを用意すること

---

### 合意事項 3: 厨房ファースト UI

**採用:** 全ての UI デザインを「厨房での使用」を第一条件として設計

**条件:**
- タッチターゲット: **最小 64px × 64px**（Material Design の 48px より大きく）
- カラーテーマ: **ダークテーマ（濃紺/黒背景）必須**
- レイアウト: **Landscape（横向き）固定前提**
- ステップ表示: **1画面 = 1ステップ**（スクロール禁止）
- フォントサイズ: **本文 18px 以上、見出し 28px 以上**
- コントラスト比: **WCAG AA（4.5:1）以上、目標 AAA（7:1）**

**Designerの納得条件:** デザインレビュー時に実際のタブレット（または同等サイズのシミュレータ）で確認すること

---

### 合意事項 4: 視聴ログの「受け皿」だけ用意

**採用:** Phase 1ではログ収集しない

**条件:**
- 型定義に `viewingLog` の構造を含める（将来の実装に備えた型のみ）
- ログ送信関数の**スタブ**（空実装）だけを用意し、将来 Supabase/Mixpanel 等を繋げる設計にする

**PdMの納得条件:** Roadmapに「Phase 2: 視聴ログとKPI可視化」を明記すること

---

### 合意事項 5: データファイルの分割設計

**採用:** 単一ファイルではなく、**ブランド別・カテゴリ別のファイル分割**設計

```
src/data/
  brands/
    unagi/
      categories.ts
      movies.ts
      sequences.ts
    ramen/
      categories.ts
      movies.ts
      sequences.ts
  index.ts   ← 全ブランドのデータをアグリゲート
```
